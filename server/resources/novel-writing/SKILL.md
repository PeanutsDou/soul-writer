# Novel Writing — 网文小说写作辅助

## 核心写作原则

### 1. 开放写作，不闭门造车

写作过程中，必要时应浏览小说网站查找对口的题材作品进行分析参考，确保对当前流行趋势和读者偏好有准确认知。

**查资料/看小说时优先使用笔趣阁镜像站 xsdi.net：**
- 访问路径：`https://m.xsdi.net/shu/{book_id}/`（章节目录页）
- 章节内容：`https://m.xsdi.net/shu/{book_id}/{chapter_id}.html`
- 该站无登录墙、无 Cloudflare 防护、无反爬验证码，比起点官网更容易获取内容
- 如需分析某部作品的具体章节，优先从 xsdi.net 获取，而非起点官网

**如 xsdi.net 未收录，降级到起点移动端 m.qidian.com：**
- 书 ID 获取：通过 `qidiantu.com/info/{book_id}/` 查询起点书籍 ID
- 目录页：`https://m.qidian.com/book/{book_id}/catalog/`（含免费/付费标记）
- 章节内容：`https://m.qidian.com/chapter/{book_id}/{chapter_id}/`
- 移动端无反爬验证码，免费试读章节可直接获取
- 注意：仅读取免费试读章节（前20章左右），不涉及付费内容

可查阅的网站还有**起点图（qidian-tu.com）**——该网站提供起点中文网每月新书的实时数据榜单，可以更直观地查看题材风向、新书表现和市场趋势。**起点图仅用于查榜单/数据，看具体内容走 xsdi.net。** 不要仅凭已有知识写作。

### 2. 语言节奏：短句优先

- **多用短句，少用长句**。一句话能说完的不要拖成三行。
- 文字要有"呼吸感"——长短句交替，快慢结合。连续短句制造紧张感，适当长句用于描写和抒情。
- 每段不宜过长，4-6行为宜。段落之间留白，给读者换气的空间。
- 避免堆砌形容词和成语，用动作和对话推进叙事。

### 3. 章节节奏：杜绝流水账

- **严禁在章节里进行流水账式叙述**。每一章必须有一个核心事件或情绪锚点。
- **严禁节奏过快**——不要恨不得一章就把一个完整的情节线跑完。给读者留出沉浸的空间。
- 合理的节奏应该是：铺垫（20%）→发展（40%）→高潮（30%）→余韵（10%）。
- 每个场景要有起承转合，不要平铺直叙一马平川。

### 4. 期待感营造

- 每一段情节都要考虑：**读者现在期待看到什么？**
- 开篇前三章必须完成：主角身份确立 + 核心矛盾浮现 + 金手指/机缘显露 + 悬念钩子。
- 每章结尾留钩子——不是硬留，而是让读者自然产生"然后呢"的冲动。
- 大悬念套小悬念：主线悬念（全书级）→ 卷悬念（10-30章级）→ 章悬念（每章结尾级）。

### 5. 文风参考

每次启动写作任务之前，必须先阅读 `reference/文风参考.md` 中的示例正文，模仿其叙述节奏、句式结构、场景转换手法。

### 6. 爽点类型库与剧情结构库

写作和构思时，查阅 reference 目录下的两个结构化工具文档：

- **`reference/爽点类型库.md`** — 收录历史分析中提取的爽点类型（认知差爽、反差爽、打脸爽、捡漏爽、格局爽、解谜爽、压迫破局爽、情绪高潮爽、情感依赖爽），每类含子类型、案例和使用建议。写作时从中选取、组合、编排爽点。

- **`reference/剧情结构库.md`** — 收录历史分析中提取的剧情模式（开篇模式、主线推进模式、三幕式分布、章节内结构等），含节奏模板和适用题材。写作时从中选取结构模板来编排情节。

**每次完成小说分析后的总结要求**：
1. 将新发现的爽点归类到 `爽点类型库.md` 中（已有分类则合并，新分类则新增）
2. 将新发现的剧情结构模式归入 `剧情结构库.md` 中
3. 两个库会随分析次数增多而稳定收敛，最终形成一套固定的类型工具箱

## 写作素材库

所有小说研究、市场分析的结果文档统一存放在 Soul Writer 项目内：
- **路径**：`server/knowledge_base/`
- **目录规范**：以 `YYYY-MM-主题` 作为一级子目录
- **文档命名**：功能命名 + 日期，如 `首订分析_2026-04-26.md`
- 做好目录索引维护，便于后续查阅

## 小说知识库

`server/knowledge_base/` 下维护了**已分析小说的结构化档案**，每本书一份，包括：

- 章节骨架（前10-20章，逐章标注核心事件/爽点/情绪/期待感）
- 原文索引（指向 `首章内容/` 目录下的前20章txt，以及 xsdi.net 链接）
- 写作技巧提取（可复用的写法和结构模式）

**使用方式**：
1. 写某个题材前，先翻 `knowledge_base/_index.md` 总索引找同类书
2. 打开对应小说的分析档案，看章节骨架和爽点设计
3. 需要精读原文时，按原文索引路径打开 txt 或 xsdi.net 链接

**维护方式**：
- 每次分析完一本新小说，在 `knowledge_base/novels/` 下新增一份 md 档案
- 同时更新 `knowledge_base/_index.md` 总索引
- 发现的爽点/结构规整到 `reference/` 下的类型库中

## 竞品分析方法

分析同类热门小说时，按以下流程进行结构化拆解：

**Step 1 — 数据筛选**
从起点图月度新书榜抓取首订数据，筛选条件：首订 3000+ + 作者等级 Lv1-Lv4 的"黑马书"。

**Step 2 — 内容拆解**
逐段阅读前三章（高潮章可到第十章），围绕三个问题分析：
- 爽点是什么？→ 分类入库（详见 `reference/爽点类型库.md`）
- 情绪怎么走？→ 记录读者情绪变化轨迹
- 期待感在哪？→ 是什么让读者想读下一章

**Step 3 — 结构提取**
画出前10章章节骨架，标注三幕式位置，分析信息章/情绪章/推进章的比例 → 归入 `reference/剧情结构库.md`

**Step 4 — 卖点提炼**
用一句话说清这本书的独特性（"系统以为这是宝可梦但其实是原创世界"，而不是"一本御兽文"）

**执行原则**：先数据后内容、不带预设、每条分析对应到具体段落、目的是提取可复用的写法。

## 全书精读：子会话并行批处理流程

当需要对一部完整小说（几百到上千章）进行系统化精读、建立全书索引时，使用以下已验证的并行批处理流程。

### 核心架构

```
主线会话（pro，监督者）→ 子会话（flash，执行者）× N并发
```

- **主线会话**：拆分批次、调度子会话、监控进度、合并结果、重试失败
- **子会话**：`claude --model deepseek-v4-flash --print` 单次调用，读批次文件→分析→写输出

### Step 1 — 章节检测与批次拆分

小说全文 txt 放在 knowledge_base 下。**不同书源的章节格式不同，必须先扫描确定模式再拆分。**

#### 章节格式分类

| 类型 | 代表作品 | 章节模式 | 编码 | 说明 |
|------|---------|---------|------|------|
| 网文中文数字 | 斗破苍穹 | `^第[零一二三四五六七八九十百千万两]+章` | UTF-8 | 字符集必须含"两" |
| 传统阿拉伯数字 | 三体 | `^第(\d+)章` | UTF-8 | 简洁，但注释章也匹配 |
| 幕章混合 | 龙族 | `^第[一二三四五六七八九十百千]+[幕章]` | GBK | 不同卷用不同结构词 |
| 多卷编号重启 | 龙族 | 每卷从第一章重启 | — | 用文件行号做唯一ID，不用章号 |

**编码检测**：先用 `utf-8` 尝试，失败则降级到 `gbk`/`gb18030`。Windows 下载的 txt 常有 GBK 编码。

#### 批次大小参考

| 章节类型 | 每章大小 | 每批章节数 | 每批大小 |
|---------|---------|-----------|---------|
| 网文（斗破苍穹） | ~5-15KB | 20-25章 | 80-200KB |
| 传统文学（三体） | ~10-15KB | 10章 | 100-150KB |
| 长章节（龙族） | ~80-120KB | 3章 | 200-350KB |

**原则**：每批控制在 200-350KB 以内。章节太长（>100KB/章）时压缩到 2-3 章/批。

#### 分析维度适配

网文和传统文学的分析维度不同，提示词需调整：

| 维度 | 网文适用 | 传统文学替换为 |
|------|---------|--------------|
| 打斗节奏 | 保留 | 叙事节奏（描写/对话/行动比例） |
| 技法标记（水字数） | 保留 | 文学技法（比喻/POV/伏笔/象征） |
| 情绪线 | 保留 | 保留 |
| 设定披露 | 保留 | 保留 |
| 钩子类型 | 保留 | 保留 |
| — | 新增 | 人物弧光 |
| — | 新增 | 主题线索 |

#### 脚本模板

核心逻辑：检测章节头 → 计算行范围 → 按批次提取原文 → 写入批次文件。

### Step 2 — 子会话提示词模板（L1+L3 合并输出）

**核心原则：一次读取，同时产出概要和精析，不读两遍浪费 token。**

```bash
# 用 Bash 工具 + run_in_background，每次 8 个并发
for i in 001 002 003 004 005 006 007 008; do
  claude --model deepseek-v4-flash --print "请读取文件 BATCH_PATH，对每个真实章节同时生成概要和精析。规则：1. 独占一行、长度小于60字符的第X章行是真实标题 2. 每章输出格式为：## 第X章 标题 换行 ### 概要 换行 200-300字叙事概要 换行 ### 精析 换行 - 情绪:xxx - 打斗节奏:xxx(如有) - 设定披露:xxx - 钩子类型:xxx - 技法标记:xxx 3. 直接写入文件 OUTPUT_PATH" &
done
wait
```

**每章输出格式（紧凑版，单章控制在 500-800 字）：**

```markdown
## 第X章 标题

### 概要
200-300字叙事概要……【情绪：xxx｜设定：xxx｜钩子：xxx】

### 精析
- **情绪线**：起(压抑)→承(蓄力)→转(反转点行号)→合(释放)
- **打斗节奏**（有打斗则标）：X段式、反转次数、视角切换频率
- **设定披露**：新设定（展示/告知/侧写）
- **钩子类型**：信息差/危机/悬念/爽感预告
- **技法标记**：重复句式、"水"字数手段、标志性修辞
```

**精析维度说明：**

| 维度 | 分析内容 | 无则省略 |
|------|---------|---------|
| 情绪线 | 情绪起点→转折→终点，标注关键句位置 | — |
| 打斗节奏 | 阶段数、反转点、视角切换、招数描写比例 | "无打斗"即可 |
| 设定披露 | 新功法/丹药/势力/规则，标注披露方式（展示/告知/侧写） | "无新设定" |
| 钩子类型 | 章末钩子分类（信息差/危机/悬念/爽感预告） | — |
| 技法标记 | 土豆标志性句式、重复用词、明显水字数段落 | 仅当明显时标注 |

**关键注意事项：**
- **并发数控制在 10 个**，既能保证速度又不会造成电脑卡顿。20+ 开始有感知卡顿，30+ 明显卡顿，60+ 系统接近卡死
- 命令行不加 `2>&1`、`&`、`$!` 等额外 shell 操作符，只用 `run_in_background: true`
- 提示词中明确输出文件路径，子会话用 Write 工具写入
- **不要用 Python subprocess 调用 claude**，Windows 上有 UTF-8/GBK 编码冲突
- 合并脚本需兼容**阿拉伯数字章号**（子会话可能将"第一千零二十章"写成"第1020章"），正则同时匹配中阿两种格式

### Step 3 — 进度监控与合并

每轮完成后用 Python 合并脚本（逐条标题去重），最终合并到主文件。

**合并脚本关键注意事项：**

1. **章号兼容两种格式**：子会话输出可能混用中文数字（"第一千零二十章"）和阿拉伯数字（"第1020章"），正则需同时匹配：
   ```python
   m_cn = re.match(r'## 第\s*([零一二三四五六七八九十百千万两]+)\s*章', title)
   m_ar = re.match(r'## 第\s*(\d+)\s*章', title)
   ```

2. **Markdown 标记干扰**：章节范围格式可能被 `**` 包裹（如 `**章节范围**:`），正则需兼容：
   ```python
   re.search(r'章节范围[*：:\s]*第?\s*(\\d+)\\s*章?\\s*[-~—–to至]+\\s*第?\s*(\\d+)\\s*章', sec)
   ```

3. **排序用提取的章号数值**，不用字符串排序（"一千"会排在"三百"前面）

### Step 4 — 失败重试

检查条件：输出文件不存在 或 大小 < 500 字节 → 单独重新调用一次。重试时用 5 个并发内的小批次。

### Step 5 — 产出清理

全书索引构建完成后，删除 `batches/` 目录及所有中间文件（`_prompt_*.txt`、`L1_*.txt`、`summary_*.md`、`analysis_*.md`、`arcs_*.md`、`L2_batch_*.txt`、`_failed.txt`）。同时删除主目录下的临时 Python/sh 脚本。

**最终保留的文件结构：**
```
knowledge_base/斗破苍穹/
├── 斗破苍穹_全文.txt        # 原始小说文本
├── _index.md                # 索引目录
├── 章节精析.md              # L1+L3 逐章分析（主产出）
├── 章节概要.md              # 早期 L1 概要（如有）
├── 单元概要.md              # L2 故事弧划分
├── 检索表.md                # 术语→章节映射
├── 检索表_原始.json         # 结构化术语数据
└── 章节索引.json            # 章节行号索引
```

### 全流程总结

从零到全书索引的一条龙流程：

```
1. 章节检测与批次拆分（Python脚本，20-25章/批）
       │
2. L1+L3 逐章精析（flash子会话 × 30并发，一次读取双产出）
       │
3. 合并到 章节精析.md（逐条去重，兼容阿/中章号）
       │
4. L2 单元概要（精析数据拆5批 → flash子会话识别故事弧）
   + 检索表（Python预提取 + AI校验补充）
       │
5. 清理中间产物，保留8个最终文件
```

**已验证的规模数据（斗破苍穹，19MB原文，~1900章）：**
- L1+L3：128批次 × 每批25章，30并发，约15分钟完成
- L2：5批次 × 每批300章精析数据，5并发，约3分钟完成
- 检索表：Python提取141术语，秒级完成
- 中间产物：641个文件（`batches/`下），清理后保留8个

## 知识库渐进式检索

完成全书索引构建后，使用 `scripts/search_kb.py` 进行三层渐进式检索。每次写作需要参考原文写法、设定细节或节奏处理时，先检索再动笔。

### 三层检索模型

```
Layer 1: 术语表命中 → 术语名、首次出场章、出场次数、关键章节
Layer 2: 弧级上下文 → 所在故事弧的章节范围、核心冲突、情绪主线
Layer 3: 章节精析 → 单章概要（200-300字）+ 五维精析
Layer 4: 原文追溯 → 按行号提取原文（仅在需要逐句分析时使用）
```

### 使用方式

```bash
# 渐进式检索（默认3层）
python scripts/search_kb.py 斗破苍穹 青莲地心火

# 只看术语匹配+弧上下文（跳过精析）
python scripts/search_kb.py 斗破苍穹 萧炎 2

# 单章完整精析
python scripts/search_kb.py 斗破苍穹 detail 389

# 单章原文（前60行）
python scripts/search_kb.py 斗破苍穹 raw 389
```

### 检索策略指导

| 写作场景 | 检索深度 | 用法 |
|---------|---------|------|
| 查某个设定/功法是否已有 | Layer 1 | `search_kb.py <书> <术语>` 看首次出场和关键章节 |
| 参考某段剧情的情绪节奏 | Layer 2 | 定位到弧后读弧概要，了解情绪主线走向 |
| 模仿某个打斗场景的写法 | Layer 3 | `detail <章号>` 看完整的五维精析 |
| 逐句拆解某个经典段落 | Layer 4 | `raw <章号>` 提取原文 |
| 关键词模糊查找 | Layer 1 | 输入部分术语名自动模糊匹配 |
| 确认某个人物的完整弧线 | Layer 1→3 | 术语表看出场章节→选关键章看精析 |

### 脚本位置

`~/.claude/skills/novel-writing/scripts/search_kb.py`

每本新书完成索引后，脚本自动可用（读取 `knowledge_base/<书名>/` 下的标准文件结构）。无需额外配置。

## 写作前准备流程

1. 阅读 `reference/文风参考.md`，建立文风参照系
2. 查阅 `reference/爽点类型库.md` 和 `reference/剧情结构库.md`，选择本段/章要使用的爽点组合和结构模板
3. （如必要）浏览目标平台同类作品——查数据看**起点图（qidian-tu.com）月度新书榜**，读内容走**笔趣阁 xsdi.net**（优先级高于起点官网）
4. 构思本段/章的核心事件、情绪锚点和期待感钩子
5. 选出适用的爽点类型，编排进情节节奏
6. 写出大纲框架（100-200字）
7. 逐段展开，注意节奏控制
